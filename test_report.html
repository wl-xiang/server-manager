#1 过期锁文件永久阻止服务启动
问题描述：当脚本在持有锁文件期间异常崩溃（或被 kill），.server.lock 文件会残留。此后所有启动操作都会失败，因为 acquire_start_lock 在检测到锁文件存在时直接返回失败，不会检查服务是否实际在运行。这意味着用户必须手动删除锁文件才能恢复服务。
# 当前代码：锁文件存在时直接返回失败，不检查服务状态 acquire_start_lock() { if ! ( set -C; : > "${LOCK_FILE}" ) 2>/dev/null; then return 1 # <-- BUG: 不检查服务是否实际运行 fi if check_pid; then release_start_lock return 1 fi return 0 }
复现步骤：
touch .server.lock && ./server start --command "python3 test_server.py"
结果：Error: Server is already running or another start is in progress
预期：应检测到锁文件过期，删除它并正常启动。
严重 逻辑缺陷 健壮性
server#L183-L233
#2 启动失败时锁文件未释放
问题描述：在 do_start 函数中，获取锁之后、释放锁之前存在多个可能失败的步骤（ensure_log_dir、rotate_log），这些步骤失败时锁文件不会被释放。一旦发生，锁文件将变为过期锁，触发问题 #1。
do_start() { ... if ! acquire_start_lock; then ... fi ensure_log_dir # 失败 → 锁泄漏 rotate_log # 失败 → 锁泄漏 : > "${LOG_FILE}" run_hook "${BEFORE_COMMAND}" # ... 启动服务 ... if ! kill -0 "${child_pid}" ... then release_start_lock # 仅此路径释放 ... fi ... release_start_lock # 成功路径释放 }
影响：如果 LOG_DIR 指向不可写路径（如权限不足），ensure_log_dir 失败后锁文件泄漏，导致问题 #1 连锁发生。

server#L329-L352
#3 do_clean 使用硬编码文件名模式，忽略 LOG_FILE_NAME 配置
问题描述：do_clean 函数中硬编码了 server*.log 的 glob 模式，而不是使用配置的 LOG_FILE_NAME。当用户设置 LOG_FILE_NAME=myapp.log 时，clean 命令无法清理自定义名称的日志文件。
do_clean() { ... for f in "${LOG_DIR}"/server*.log; do # <-- 硬编码 "server" [ -f "$f" ] && count=$((count + 1)) done ... rm -f "${LOG_DIR}"/server*.log # <-- 同样硬编码 }
复现：LOG_FILE_NAME=myapp.log ./server start ... && ./server clean → 输出 "No log files to clean"，但 myapp.log 未删除。
建议：应使用 ${LOG_FILE_NAME} 或基于 ${LOG_FILE_NAME} 构建 glob 模式。
中等 逻辑缺陷
server#L160-L171
#4 LOG_MAX_BACKUPS=0 时仍然创建备份文件
问题描述：当 LOG_MAX_BACKUPS=0 时，rotate_log 仍然会执行 mv "${LOG_FILE}" "${LOG_FILE}.bak.1"，创建了不应该存在的备份文件。设置 0 的语义应该是"不保留任何备份"，但实际行为是"保留 1 份备份"。
rotate_log() { [ -f "${LOG_FILE}" ] || return 0 rm -f "${LOG_FILE}.bak.${LOG_MAX_BACKUPS}" # rm server.log.bak.0 (无害) i=$((LOG_MAX_BACKUPS - 1)) # i = -1 while [ "${i}" -ge 1 ]; do ... done # 不执行 mv "${LOG_FILE}" "${LOG_FILE}.bak.1" # <-- BUG: 总是创建 bak.1 }
复现：LOG_MAX_BACKUPS=0 ./server start ... && ./server restart ... → server.log.bak.1 文件存在。

#5 选项缺少参数值时崩溃（shift 错误）
问题描述：parse_start_options 和 do_log 函数在选项（-c、-b、-a、-n）缺少参数值时，shift 2 会因为参数不足而报错 "shift: can't shift that many"，脚本退出码为 2。
# 触发方式： ./server start -c # -c 需要参数值，但没提供 ./server logs -n # -n 需要参数值，但没提供 # 错误输出： ./server: 81: shift: can't shift that many
影响：用户体验差，且退出码 2 不是预期的 1（一般错误）。应检查参数数量后再 shift。

server#L160-L171
#6 LOG_MAX_BACKUPS 值未做合法性校验
问题描述：LOG_MAX_BACKUPS 接受任意值（包括非数字、负数），不做校验。设置 LOG_MAX_BACKUPS=abc 时，$((abc - 1)) 在 POSIX sh 中会产生算术错误，但脚本继续执行并产生不可预期的行为。
建议：在启动时验证 LOG_MAX_BACKUPS 是否为非负整数。

#7 shellcheck SC2295: ${var#pattern} 中 pattern 未正确引用
问题描述：在 load_env 函数的引号剥离逻辑中，${_val#${_dq}} 和 ${_val%${_dq}} 中的 ${_dq} 未加引号，会被当作 glob 模式匹配而非字面字符串。
# 当前 (有问题) _val="${_val#${_dq}}"; _val="${_val%${_dq}}" # 建议修改为 _val="${_val#"${_dq}"}"; _val="${_val%"${_dq}"}"
影响：虽然引号字符 " 和 ' 不是特殊的 glob 字符，但不符合 POSIX 最佳实践，在某些极端 shell 实现中可能有问题。

server#L283, L286
#8 shellcheck SC2059: printf 格式字符串中含变量
问题描述：do_status 函数中 printf 的格式字符串包含了 ${COLOR_GREEN} 等变量，不符合 shellcheck 规范。如果颜色变量中包含 % 字符，可能导致意外行为。
# 当前 (有问题) printf "Server is ${COLOR_GREEN}running${COLOR_RESET} (PID: $(get_pid))\n" # 建议 printf 'Server is %srunning%s (PID: %s)\n' "${COLOR_GREEN}" "${COLOR_RESET}" "$(get_pid)"

server#L119-L128
#10 PID 文件仅验证进程存在，不验证进程身份
问题描述：check_pid 使用 kill -0 仅检查 PID 是否存在，不验证该进程是否确实是本脚本启动的服务。如果服务进程崩溃后 PID 被系统回收并分配给其他进程，脚本会误认为服务仍在运行，并尝试向该进程发送 SIGTERM。
风险：低（PID 回收在 Linux 上需要较长时间且通常不会立即分配给其他进程），但理论上存在。
轻微 健壮性
server#L218
#11 sleep 1 硬编码可能导致误判启动失败
问题描述：do_start 中启动后仅等待 1 秒就检查进程是否存活。对于初始化时间超过 1 秒的服务，可能被误判为启动失败。
# 当前硬编码 sleep 1 if ! kill -0 "${child_pid}" 2>/dev/null; then print_error "Error: Failed to start server" fi
建议：可配置的等待时间，或采用重试机制。
轻微 安全
server#L174-L177
#12 run_hook 使用 eval 执行钩子命令
问题描述：run_hook 使用 eval 执行 BEFORE_COMMAND 和 AFTER_COMMAND。如果这些命令来自不可信来源（如被篡改的 .env 文件），可能造成任意代码执行。
run_hook() { [ -n "$1" ] || return 0 eval "$1" >> "${LOG_FILE}" 2>&1 # eval 执行任意命令 }
风险：低（需要先能修改 .env 文件或环境变量），但值得注意。
轻微 用户体验
server#L106-L111
#13 环境变量优先于命令行参数的设计违反直觉
问题描述：resolve_commands 中环境变量优先级高于命令行参数（ENV_START_COMMAND:-${CLI_START_COMMAND}}），这与大多数 CLI 工具的惯例相反。用户通过命令行指定的参数被环境变量静默覆盖，可能导致困惑。
# 当前：ENV 优先 START_COMMAND="${ENV_START_COMMAND:-${CLI_START_COMMAND}}" # 示例：命令行参数被忽略 START_COMMAND="python3 app.py" ./server start --command "python3 other.py" # 实际执行的是 app.py，而非 other.py
注意：README 中已记录此行为，属于设计决策。但容易导致用户困惑。

最严重的问题：
